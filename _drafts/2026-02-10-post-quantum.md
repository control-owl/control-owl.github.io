# Abstract

This document describes the architecture, threat model, and design rationale for a multi‑source entropy pipeline used to generate **hierarchical deterministic (HD) wallet master keys**. The system combines external QRNG data, CPU hardware RNG, local jitter entropy, and a post‑quantum KEM shared secret. These inputs are mixed and processed through SHA‑3/512 to generate a high‑quality seed for **BIP32/BIP44 master key derivation**.

The purpose of this design is to guarantee that the **final master private key is secure as long as at least one entropy source remains unpredictable**, even if other components degrade, leak, or are compromised.  
The produced master keys support **both secp256k1 and ed25519** coin families.


# 1. Introduction

Modern wallet systems rely heavily on the quality of their entropy, because the entire structure of an HD wallet — including every derived address and private key — depends on the security of the **single master private key**.

If entropy fails, the whole wallet fails.  
If entropy is predictable, every coin derived from BIP32/BIP44 becomes predictable as well.

This document describes a system designed to avoid that scenario by combining independent entropy sources, using post‑quantum components, and applying a strong extraction function before generating the **seed used to create BIP32 master keys**.

This design introduces a multi‑input entropy pipeline that:

- **does not trust any single entropy provider**
- incorporates post‑quantum assumptions
- remains robust even under partial compromise
- resists silent, long‑term entropy failure
- produces uniform, high‑quality seeds for key derivation

The system’s core requirement is straightforward:

> **If any one entropy source remains unknown to the attacker, the final derived key remains unpredictable.**


# 2. Security Objectives

### 2.1 Primary Objective

Guarantee that the **HD master private key** generated by the app remains secure under classical and quantum‑assisted adversarial models.

### 2.2 Secondary Objectives

- Ensure security of all derived keys (BIP32/BIP44) for secp256k1 and ed25519.
- Eliminate single points of failure in entropy sources.
- Maintain uniformly distributed seed material.
- Resist silent failures in entropy components.
- Provide long‑term protection even if some entropy sources are later compromised.


# 3. Assets Requiring Protection

This system protects:

- Raw entropy inputs
- Mixed entropy pool
- Post‑extraction seed material
- **BIP32/BIP44 master private key**
- All chain codes
- All child private keys (both hardened and non‑hardened)
- Derived addresses for secp256k1 and ed25519 coins
- Any signing keys used by the wallet (if applicable)

Preserving the secrecy of the **master private key** is critical, because every derived key depends on it.


# 4. Threat Model

## 4.1 Attacker Capabilities

### External capabilities

The adversary may:

- Record all network traffic indefinitely.
- Log responses from remote QRNG providers.
- Modify, reorder, or inject packets.
- Conduct timing‑analysis attacks.
- Apply quantum‑accelerated brute force (Grover).
- Accumulate historical artifacts for future PQ attacks.

### Internal capabilities

The adversary may:

- Compromise a QRNG backend.
- Influence or backdoor CPU hardware RNG (IntelME).
- Reduce or partially control jitter entropy.
- Read portions of device memory.
- Recover partial DRBG internal state.
- Perform advanced side‑channel monitoring.

## 4.2 Attacker Limitations

The design assumes the adversary:

- Cannot break SHA‑3/Keccak‑512.
- Cannot compromise **all entropy sources simultaneously**.
- Cannot break lattice‑based cryptography (Kyber/Frodo/NTRU).
- Cannot eliminate device‑internal timing jitter.
- Cannot Grover‑search AES‑256 within feasible limits.

These boundaries define a realistic yet adversarial security environment.


# 5. System Architecture

## 5.1 Entropy Sources

### 5.1.1 External QRNG (Untrusted by Default)

An external quantum RNG provides high‑quality randomness, but is always treated as potentially logged or compromised.

### 5.1.2 CPU Hardware RNG

Built‑in RNG instructions (RDRAND/ RDSEED / ARM TRNG).  
Possible vulnerabilities: reduced output quality, firmware backdoors.

### 5.1.3 Jitter Entropy (Local Micro‑Timing Noise)

Collected from CPU timing variations. Hard for remote attackers to influence or observe.

### 5.1.4 PQ‑KEM Shared Secret

The shared secret from a post‑quantum KEM round is incorporated as a fourth entropy contributor.

---

## **5.2 Entropy Pipeline**

```
(External QRNG
CPU Hardware RNG
Jitter Entropy
PQ-KEM Shared Secret)
    ↓
XOR Mixer
    ↓
SHA3-512 Extractor
    ↓
Seed
    ↓
Master Private Key
    ↓
Derived Addresses & Keys
(secp256k1 & ed25519)
```

### **5.2.1 XOR Mixing**

All entropy sources are XOR-ed together.  
Security principle: **if any input remains unknown, the final XOR result is unknown.**

### **5.2.2 SHA‑3/512 Extraction**

The output of the mixer is run through SHA‑3/512.  
This produces:

- uniform distribution
- bias removal
- domain separation
- resistance to preimage attacks

### 5.2.3 Seed Output

The output of SHA‑3/512 becomes the **entropy basis for HD wallet key generation**.

- For **BIP32/BIP44**, the extracted seed is fed directly into standard master key derivation.
- From that master key, all hardened and non‑hardened child keys are derived.
- The pipeline supports both **secp256k1** and **ed25519** derivation paths.


# 6. Key Derivation Process

1. **SHA‑3/512 extractor output** becomes the wallet seed.
2. The seed is passed through the standard functions defined in:
    - **BIP32** (Hierarchical Deterministic Wallets)
    - **BIP44** (Multi‑account structure)
3. The result is a **master private key + chain code**.
4. From this master key, the application can derive:
    - secp256k1 keys (Bitcoin‑like chains)
    - ed25519 keys (Solana)
5. All coin addresses and private keys follow the standard HD path, e.g.:

```
m / 44' / coin_type' / account' / change / index'
```

# 7. Deterministic Random Bit Generator (DRBG)

The DRBG is responsible for generating:

- nonces
- ephemeral keys
- per‑session identifiers
- randomness required by protocols

### Key DRBG Properties

- 512‑bit internal state
- continuous state updates
- periodic reseeding from the entropy pipeline
- resilience against partial state compromise
- halting or refreshing behavior when entropy degrades

It continues to provide nonces, ephemeral randomness, and protocol identifiers even after wallet creation.


# 8. Failure and Degradation Handling

The pipeline is designed to tolerate:

- silent hardware RNG degradation
- partial compromise of remote QRNG
- timing entropy reduction from environmental changes
- partial leakage of PQ‑KEM secrets
- temporary loss of one or more entropy inputs

The essential safety property is:

> Wallet seeds remain secure if **any** entropy source remains unpredictable.

This condition is intentionally difficult to achieve.


# 9. Post‑Quantum Considerations

### 9.1 Symmetric Algorithms

AES‑256 maintains ~128 bits of quantum‑resistant security after Grover.

### 9.2 Hash Functions

SHA‑3/512 provides significant margin against quantum‑accelerated preimage attacks.

### 9.3 Lattice‑Based KEM

Kyber/Frodo/NTRU shared secrets add cryptographic hardness independent of device entropy.

### 9.4 Signatures

Multiple PQ signature schemes will be evaluated to avoid reliance on a single algorithm family. The PQ‑KEM secret contributes directly to the seed that feeds the BIP32/BIP44 master key process, giving an extra security layer even if classical entropy is partially compromised.


# 10. Security Summary

- HD wallets rely on a **single master key**; entropy quality must be maximal.
- Multi‑source entropy mixing protects against single‑point RNG failures.
- SHA‑3/512 ensures uniform seed distribution for BIP32/BIP44.
- secp256k1 and ed25519 master keys are both supported.
- Quantum threats are considered in the initial seed generation.
- The final master key is unpredictable unless **all** entropy sources are compromised.

This architecture is intended to offer long‑term safety under strong adversarial conditions without relying on ideal behavior from any individual component.

# **11. Conclusion**

This architecture provides a secure method for generating **HD wallet master private keys** using post‑quantum‑aware entropy mixing. The integration of multi‑source entropy, SHA‑3 extraction, and PQ‑KEM secrets creates a foundation that protects all coins, keys, and addresses derived under BIP32 and BIP44 — for both secp256k1 and ed25519 ecosystems.
